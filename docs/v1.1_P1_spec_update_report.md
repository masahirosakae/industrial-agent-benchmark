### 3.2 採点優先順位 (確定値)

ユーザー要件に従い、以下の **厳密な順序** を Section 9.1 / 9.8 に明記した。

```
優先順位 (高 → 低):
  1. critical_failures
  2. disallowed_answers
  3. score_cap_rules
  4. numeric_checks            (required: true 失敗は critical 相当に昇格)
  5. must_have
  6. structured_output_requirements (format パース不可は critical 相当に昇格)
  7. generic_answer_penalty
  8. nice_to_have
  9. reasoning_quality
```

### 3.3 最終スコア算出式

```
final_score = max( min(matrix_score, min(caps)) + penalty , floor )
```

ただし、(1) critical_failures、(2) disallowed_answers、
(4) numeric_checks.required 失敗、(6) structured_output の format パース不可のいずれかが
発火した場合、上式をスキップして `final_score = 1` を即時返す (早期確定経路)。

- 本章は `schema_version: "1.1"` の問題にのみ適用される
- v1.0 既存 90 問は Section 1〜8 のみで採点され、本章の影響を受けない
- スコアレポートは v1.0 / v1.1 を同一指標 (1〜5) として扱えるが、
  カテゴリ別 / schema_version 別の分布を併記することを推奨

### 4.2 検証規則 (v1.1 任意フィールド)

各フィールドは **存在する場合のみ** 検証する (未指定は OK)。
ユーザー指定要件への対応:

| ユーザー要件 | 実装 |
|---|---|
| v1.0 問題は従来どおり通る | `validate_v11_extensions` は v1.1 フィールド未指定時に no-op (`empty_v10` テストで確認) |
| v1.1 追加フィールドは任意 | 各 `if "<field>" in data:` で囲ってある |
| 存在時は構造検証 | 下記参照 |
| `score_cap_rules` あり → `condition` / `max_score` 必須 | ✅ `condition` 非空 string, `max_score` int 1〜5 を必須 |
| `numeric_checks` あり → `name` / `expected_value` / `tolerance` / `unit` 必須 | ✅ 4 つすべて必須化、`required` のみ任意 (型は bool) |
| `structured_output_requirements` あり → `required` / `format` / `required_fields` を検証 | ✅ `required` bool, `format` enum, `required_fields` list[str]。`required=true` のとき `required_fields` 非空必須 |
| `generic_answer_penalty` あり → `enabled` / `conditions` を検証 | ✅ `enabled` bool, `conditions` list[str]。`enabled=true` のとき `conditions` 非空必須。標準条件以外は **warning** |

加えて以下も検証する:

- `schema_version` あり → `"1.0"` / `"1.1"` のみ許可
- `score_cap_rules[*].reason` あり → string 必須 (任意フィールド)
- `numeric_checks[*].expected_value` → number / string (リスト等は拒否)
- `disallowed_answers` あり → list[非空 string]

### 4.3 後方互換性の担保方針

- **トリガ条件**: `if "<field>" in data:` でフィールド存在チェック。
- **未指定問題**: v1.0 90 問はいずれの v1.1 フィールドも持たないため、`validate_v11_extensions` は実質 no-op。
- **エラー/警告**: v1.1 フィールド未指定の問題に対しては一切エラー / 警告を出さない (`empty_v10` テストで確認)。
- **設計判断**: v1.1 フィールドの構造検証は `schema_version` 値に依らず発火する。
  これは「v1.0 問題に v1.1 フィールドが誤って混入した場合も構造的に検出する」ためのフェイルセーフ。
  v1.1 問題に v1.1 フィールドを必須化することは P1 では行わず、P2 で問題生成後に必要なら追加する。
