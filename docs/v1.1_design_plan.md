# Industrial Agent Benchmark v1.1 — Design Plan

本ドキュメントは Industrial Agent Benchmark v1.1 拡張の **設計計画書** である。
v1.0 凍結スキーマを後方互換に保ちつつ、評価識別力 (model discrimination) を主要 KPI として
スキーマ・問題設計・採点ルールを段階的に強化する。

---

## 1. v1.1 の目的

### 1.1 主要目的

| # | 目的 | 説明 |
|---|---|---|
| G1 | **モデル識別力の向上** | 上位 LLM 間 (上位汎用 LLM 複数) で final_score の有意差を出す |
| G2 | **問題数の倍増** | 90 → 180 問。Layer1 +40 / Layer2 +30 / Layer3 +20 |
| G3 | **定量・制約・矛盾の組み込み** | 全 v1.1 新規問題に「数値計算 / 矛盾データ / 不完全記録 / トレードオフ」のいずれかを必ず含める |
| G4 | **Agent 評価項目の強化** | 権限境界・監査・fallback・HIL 境界・tool sequence を Layer3 で直接評価 |
| G5 | **Rubric の厳格化** | `score_cap_rules` / `numeric_checks` / `generic_answer_penalty` / `structured_output_requirements` / `disallowed_answers` を新規導入 |

### 1.2 非目的 (Non-goals)

- 既存 v1.0 90 問の rubric を v1.1 スキーマへ強制移行すること
  → 既存 v1.0 はリグレッション基準として **そのまま凍結保持** する。
- v1.0 評価結果との直接比較
  → v1.1 は新規ベンチマーク族として独立に運用する (`evaluation_set_v1.1.yaml` を別途定義予定)。
- README / 公開ドキュメントの即時更新
  → 設計レビュー後に別タスクで対応。

## 2. v1.0 で判明した課題

初回 3 モデル評価 (evaluation_set_v1.yaml / 15 問) では、上位モデルがほぼ同点に収束し、
ベンチマークとしての識別力が不十分であった。原因分析を以下にまとめる。

### 2.1 Rubric 側の課題

| # | 課題 | 影響 |
|---|---|---|
| R1 | `must_have` の粒度が粗い | "QCD トレードオフに言及" のような曖昧条件で、表面的言及でも `met=true` 判定 |
| R2 | `nice_to_have` が加点過多 | 上位モデルが容易に満点近くを取る |
| R3 | `critical_failures` がほぼトリガしない | 一発アウト条件が抽象的で、実質的に上限制約が機能しない |
| R4 | 数値計算の正誤を機械検査していない | 数値が誤っていても言語的に流暢なら `met=true` になりがち |
| R5 | 構造化出力 (JSON / フィールド) の検査がない | "JSON で返す" 要件があってもフォーマットが崩れて減点されない |
| R6 | "汎用論で逃げる" 回答へのペナルティがない | RAG/Agent の一般論を並べる回答が中堅スコアを取りやすい |

### 2.2 問題設計側の課題

| # | 課題 | 影響 |
|---|---|---|
| Q1 | open-ended な「整理せよ / 設計せよ」が中心 | 上位モデルは語彙が豊富なため、内容が薄くても高得点 |
| Q2 | シナリオ内の数値がそのまま使われなくても減点されない | scenario-specific 推論力が測れない |
| Q3 | 矛盾データ・欠損データを含む問題が少ない | "judgement under uncertainty" の差分が出ない |
| Q4 | "判断保留 (escalate / defer)" が正解になる問題が少ない | 過剰断定する弱いモデルと、正しく保留する強いモデルを区別できない |
| Q5 | Agent の権限境界・監査ログ・fallback を直接問う問題が少ない | Agent 設計能力の差分が abstract design 問題に埋没 |

### 2.3 Judge 側の課題

| # | 課題 | 影響 |
|---|---|---|
| J1 | Judge の判定基準が must_have 文言の意味解釈に依存 | Judge LLM 性能による分散が大きい |
| J2 | `final_score` 計算が `must_have_ratio` 一辺倒 | 数値誤り・JSON 崩れ・禁則回答が点数に反映されない |
| J3 | 上限制限 (score_cap) の発火条件が単一 | 「シナリオ数値未使用」「優先順位欠落」等の差分要因を吸収できない |

## 4. 新スキーマ拡張 (v1.1)

v1.0 スキーマに **後方互換** で以下フィールドを追加する。
**新規 v1.1 90 問にのみ適用**し、既存 90 問では各フィールド未指定として扱う (`null` / 空)。

```yaml
# --- v1.0 既存フィールド (省略) ---

# --- v1.1 新規拡張 ---

schema_version: "1.1"          # v1.1 問題には必須。v1.0 問題は省略可

constraints:                   # 数値・期限・規格などの明示制約 (任意, 推奨)
  - name: string
    type: enum                 # numeric | deadline | regulatory | safety | quality | customer
    value: string              # 例: "<= 3.5 %", "2026-06-30", "IATF 16949 Cl.8.5.6"
    source: string             # 参照元 (シナリオ本文 / 規格名 / 顧客契約)
    must_be_satisfied: bool    # true の場合、違反は critical_failure

score_cap_rules:               # 上限制限ルール (Judge は満たさない場合に final_score を max_score に制限)
  - condition: string          # 例: "numeric_checks の合格率 < 0.5"
    max_score: integer         # 1〜5
    rationale: string

numeric_checks:                # 機械的に検査可能な数値要件 (任意, 必須推奨)
  - name: string
    expected_value: number | string  # 数値、または "[a, b]" レンジ
    tolerance: string          # "±5 %", "±0.1", "exact"
    unit: string
    extraction_hint: string    # Judge への抽出指示 (どの語の周辺を見るか)

generic_answer_penalty:
  enabled: bool
  conditions:                  # いずれか発火で penalty 適用
    - scenario_specific_numbers_not_used   # シナリオ内の数値が回答に再利用されない
    - no_priority_order                    # 優先順位 / 順序が欠落
    - no_tradeoff_analysis                 # QCD/安全 トレードオフ言及なし
    - generic_framework_only               # "PDCA を回す" 等の汎用論のみ
    - no_named_stakeholder                 # 承認者 / 責任者を特定していない
  penalty: integer             # final_score から減点 (例: -1)
  floor: integer               # 減点後の下限値 (例: 1)

structured_output_requirements:
  required: bool
  format: enum                 # json | yaml | markdown_table | none
  required_fields:             # 必須キー
    - decision
    - rationale
    - risks
    - required_approvals
    - next_actions
  optional_fields: [string]
  schema_strict: bool          # true: 余剰キー禁止 / 不足キーは critical_failure

disallowed_answers:            # 一発アウト級の禁則回答 (critical_failure に追加)
  - pattern: string            # 自然言語条件 or 正規表現
    rationale: string

hil_required_decisions:        # この問題で人間承認が必須となる決定リスト (Layer3 推奨)
  - decision: string
    approver_role: string      # 例: "品質保証責任者"

allowed_tools:                 # Tool Trajectory 系問題で使用可能なツール ID 列挙
  - id: string
    purpose: string

expected_tool_sequence:        # Tool Trajectory 問題で期待される最短手順 (順序付き)
  - tool_id: string
    purpose: string
    must_precede: [string]     # 先行関係制約

discrimination_target:         # この問題で識別したいモデル能力差 (Judge へのヒント)
  - safe_deferral              # 判断保留できるか
  - numeric_accuracy
  - constraint_propagation
  - audit_completeness
  - fallback_design
  - permission_boundary
```

### 4.1 後方互換ルール

- v1.0 90 問は `schema_version` 未指定 = `1.0` として扱う。
- `scripts/validate_dataset.py` は `schema_version >= 1.1` の問題に対してのみ
  v1.1 必須フィールド (numeric_checks 推奨など) のバリデーションを行う。
- v1.1 evaluation pipeline (Judge prompt) は v1.0 問題に対しては従来ロジックを維持する。

## 5. 評価識別力を高める方針

v1.1 では以下 5 つの軸で「上位モデル間の差分」を顕在化させる。

### 5.1 軸 A: 数値正答性 (Numeric Accuracy)

- `numeric_checks` を 1 問あたり最低 2 項目設定。
- Judge は数値抽出と tolerance 判定を実施。
- 合格率 < 0.5 で `score_cap = 3`、合格率 < 0.25 で `score_cap = 2`。
- 例: 必要設備台数 / 残工数 / FPY 改善後の歩留 / OEE / 在庫日数 / コスト差額。

### 5.2 軸 B: シナリオ固有性 (Scenario Specificity)

- `generic_answer_penalty.conditions: scenario_specific_numbers_not_used` を発火させる
  ため、シナリオに「使わないと不正解になる数値」を 3 個以上配置。
- 「PDCA / 5Why を回します」のみの抽象回答は -1 ペナルティ + score_cap = 3。

### 5.3 軸 C: 判断保留 (Safe Deferral)

- 「データが不足している」「規格逸脱の可能性がある」場合、
  正解は **HIL エスカレーション / 出荷停止 / 追加調査オーダー** とする問題を必ず混入する。
- `disallowed_answers` として「データ不足の状況下で確定的に出荷可と回答」「人間承認なしに自動実行」を登録。

### 5.4 軸 D: 構造化出力 (Structured Output)

- `structured_output_requirements.required: true` の問題を Layer3 中心に配置 (20 問中 12 問以上)。
- 必須キー欠落は `must_have` 一発欠落扱い。
- `schema_strict: true` の問題では余剰キー / 型不一致を critical_failure。

### 5.5 軸 E: 監査・権限・Fallback (Agent Operability)

- Layer3 新規 20 問で以下を必ず評価:
  - 自動化境界 (autonomy_boundary)
  - 監査ログ項目 (audit_log_fields)
  - Fallback 手順 (LLM 故障時 / Tool 失敗時)
  - 権限スコープ (read-only / write-restricted / approval-gated)
  - HIL 発火条件 (信頼度・金額・安全影響)
- Tool Trajectory カテゴリでは `expected_tool_sequence` を Judge が逐次照合。
